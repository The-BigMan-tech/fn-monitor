Unlike static analysis,u cant dig into a node to get the child scope nor execute it.the interpreter has to execute each node and discover each scope as they come to keep the program stable as it is also running it simultaneously.so u can only dig into the node to get info about its children as a pure data structure.but u can climb up to get the scope of the parent since that one has been discovered first.and with the parent scope and the name of the node if it exists,u can retrieve any object made in the past

so the flow of the icm is to preprocess ur code with generated data based on ur capture and inline dependencies,parse ur code,cache the ast then on each run---it runs a small cached ast to call the fn,encounters a node,calls ur listener with the node,scope and execute fn,then implicitly executes the same mutable node if u already havent

since each scope and node sval creates is a fresh new object even for the same ast when revisited,doing this means that each event object is isolated cuz they keep they dont keep a live ref to the current node and scope but only the ref at the time it was made.this means that the current node and scope can safely be reassigned without affecting them.but for visit,i still make the visist reflect the current data by only creating one visit instance per interpreter and making it use lazy properties.

but because of the execution flow,if u mutate an event node and scope within a listener,it will affect the execution of the node.but if u save the event and do it after,it wont  because its pointing a past ref.do semantics like this make the monitor reliable against undefined behaviour

The normalized evaluator is only called by sval for sync code while the generator-evaluator is called for non-sync code like async and generators.this means that usually,the generator-evaluator only needs generator listeners to handle its full features while the normalized one only needs the a regular function listener to work with its full features.But because i now support leveraging the two evaluators in the same listener for seamless interoperability between evaluators and the listeners and ease of use,i added generator management in the regular evaluator and non-generator management in the generator-evaluator.I dont rememer the exact details of my thoughts when i made the evaluators but it was something very similar to what i said
